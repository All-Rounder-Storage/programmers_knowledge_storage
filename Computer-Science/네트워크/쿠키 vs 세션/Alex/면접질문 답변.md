## 1. 쿠키와 세션은 언제 사용해야 적합한지 비교 설명해주세요.
```markdown
쿠키는 브라우저 내에 데이터를 저장하기 때문에 탈취, 변조가 가능하여 보안에 취약합니다
따라서 사용자의 중요한 정보를 쿠키에 넣는 것은 적합하지 않습니다
쿠키를 사용하는 방식은 예시로 팝업창의 오늘 다시보지 않기의 여부나, 비로그인 시 장바구니 기능 등 탈취나 변조를 당해도 크게 문제 없는 데이터를 쿠키에 저장하는 것이 좋습니다
반대로 탈취, 변조가 되어서는 안되는 사용자의 비교적 중요한 데이터는 모두 세션에 저장을 해야합니다.
따라서 사용자 로그인 인증 같은 여부는 쿠키보다는 세션을 이용하여 기능을 구현합니다
```
  
## 2. 쿠키와 세션을 이용한 로그인 방식을 설명해주세요.
```markdown
HTTP 특성 중 무상태(Stateless) 특징으로 인해 서버는 매 요청마다 로그인 인증을 하여야합니다.
이는 비효율적이기 때문에 쿠키/세션을 통한 로그인 인증 방식을 따로 저장해 놓을 수 있습니다.

1. 클라이언트에서는 로그인을 진행합니다.
2. 서버에서는 로그인 인증을 한 뒤 인증에 성공하면 해당 사용자의 인증 정보를 세션에 저장합니다.
    - 여기서 세션은 서버 리소스를 많이 잡아먹기 때문에 저장할 데이터를 최소화 합니다.
3. 세션에 저장하며 생긴 고유 ID인 Session ID를 쿠키에 저장합니다.
4. 반환 데이터와 함께 쿠키를 반환합니다.
5. 사용자가 다른 페이지를 재요청 하였을 때 서버는 쿠키에 있는 Session ID를 통해 Session DB에서 사용자를 확인합니다.
```

## 3. 쿠키와 세션에 대해 설명해주세요. (N사 면접질문)
```markdown
먼저 쿠키와 세션은 서로 상반되는 개념이거나 대체되는 기술이 아닌 서로 상호보완적인 관계입니다
쿠키와 세션은 HTTP 프로토콜의 특성인 Stateless를 보완하기 위해 나온 기술로써 사용자의 정보를 저장하는 저장소입니다
매 요청마다 사용자 인증을 진행하기에는 너무 비효율적이니까요
쿠키와 세션의 차이점은 크게 4가지로 구분됩니다
저장위치, Lifecycle, 용량제한, 보안 등이다

쿠키는 브라우저에 저장을 하고 세션은 서버에 저장을 한다

쿠키는 만료시간을 설정하면 직접 삭제하지 않는 이상 브라우저가 종료되도 살아 남아있다
하지만 세션은 만료시간을 설정하더라도 창닫기를 통해 브라우저를 종료하면 자동으로 삭제 된다

쿠키는 클라이언트 당 최대 300개, 도메인당 20개, 한 쿠키당 4KB의 저장 제약이 있지만
세션은 서버의 용량이 버텨주는 한 저장 제약이 없다
세션은 하나의 클라이언트(사용자)가 웹 사이트의 머무는 동안 진행하는 모든 요청에 대해서 하나의 세션으로 기준을 잡기 때문에 클라이언트 하나당 세션 하나라고 생각하면 된다

쿠키는 브라우저에 저장되기 때문에 탈취, 변조에 위험이 있어 보안에 취약하지만
세션은 브라우저에는 Session ID만 저장하고 데이터는 서버에 저장하므로 비교적 안전하다.


```
